Μεταγλώττιση και Εκτέλεση

Πριν τη μεταγλώττιση του προγράμματος εκτελέστε το install_boost.sh:

$ sh ./install_boost.sh

Το script θα εγκαταστήσει την βιβλιοθήκη “program_options” της Boost τοπικά (μέσα σε subdirectory του ./libs/), έτσι ώστε με τη διαγραφή του directory της εργασίας να διαγραφεί και η ίδια. Το βήμα αυτό μπορεί να παραληφθεί αν το σύστημα της εγκατάστασης έχει προεγκατεστημένη την παραπάνω βιβλιοθήκη (boost1_68) και τόσο το .so αρχείο όσο και τα αντίστοιχα headers βρίσκονται στις default τοποθεσίες ώστε να αναγνωριστούν από τον g++.

Για να μεταγλωττίσετε το πρόγραμμα, απλά πληκτρολογείστε

$ make

Το οποίο θα δημιουργήσει τα εκτελέσιμα lsh και cube στο παρόν directory. Εναλλακτικά, εκτελέσετε

$ make lsh

ή

$ make cube

για τη δημιουργία μόνο του ενός εκτελέσιμου.

Η εκτέλεση των παραπάνω αρχείων γίνεται με τις παραμέτρους που προσδιορίστηκαν στην εκφώνηση. Μπορείτε να τα τρέξετε με όρισμα --help για να αναλυτική περιγραφή των ορισμάτων της γραμμής εντολών.

Λεπτομέρειες σχεδίασης και υλοποίησης

Ο κώδικας υλοποιήθηκε με τη χρήση της C++ (standard 2011). Χρησιμοποιήθηκε η STL για τις περισσότερες εσωτερικές δομές και η βιβλιοθήκη Boost μόνο για να διευκολύνει το διάβασμα των ορισμάτων της γραμμής εντολών. Θεώρησα πως το σημείο αυτό είναι πολύ δευτερεύον γι αυτό και χρησιμοποίησα 3rd party library.
Γενικά χρησιμοποιήθηκε αντικειμενοστραφής σχεδιαση με τη χρήση κληρονομικότητας και abstract interfaces.

Κατά τη διάρκεια της γραφής του κώδικα, χρησιμοποιήθηκε το git ως version control system και το GitHub ως remote repository. Μάλιστα, χρησιμοποιήθηκαν εκτενώς τα λοιπά feature που προσφέρει το GitHub (Pull Requests, Issues, Projects). Επίσης μεγάλο μέρος του κώδικα καλύπτεται από unit tests (χρησιμοποιώντας το google test), ωστόσο δεν θα έλεγα πως έγινε κανονικό “test driven development”.

Σημεία του κώδικα που αξίζει να επισημανθούν:

  • Υλοποιήθηκε η κλάση NDVector ώστε να αναπαριστώνται διανύσματα του n-διάστατου χώρου. Αν και θα μπορούσα να έχω χρησιμοποιήσει το std::vector<double> για την ίδια δουλειά θεώρησα πως είχε νόημα να γράψω συγκεκριμένη κλάση. Επίσης χρησιμοποιήθηκε operator overloading για να υλοποιηθούν οι συνιθισμένες διανυσματικές πράξεις (οι οποίες βέβαια, πέρα από το εσωτερικο γινόμενο και την ευκλείδεια νόρμα δεν χρειάζονται).
  • Τα dataset αναμένεται να έχουν τη μορφή της εκφώνησης (δηλαδή η πρώτη στήλη να περιέχει το item_id και οι στήλες να είναι χωρισμένες με ΤΑΒ). Τα item_id αποθηκεύονται ως strings εσωτερικά στο πρόγραμμα. Είναι δυνατό να περιέχονται κενές γραμμές στην αρχή του κάθε αρχείου.
  • Αφού γίνει ο υπολογισμός του g(p) στο ευκλείδειο LSH, για την εύρεση της θέσης του πίνακα στο hash table δε χρησιμοποιείται η φ(p) όπως προείνεται στην εκφώνηση. Αντίθετα, μετατρέπεται το g(p) σε string και χρησιμοποιείται η default hash function της STL.
  • Κατά την εκτέλεση του προγράμματος εκτυπώνονται μερικά μηνύματα προόδου, τα στατιστικά που ζητούνται καθώς και μερικά ακόμα σχετικά στατιστικά. Παράδειγμα εκτέλεσης:

Βελτιστοποιήσεις

  • Στο κομμάτι του ευκλείδειου LSH, η “υπερπαράμετρος” w επηρεάζεται σε μεγάλο βαθμό από τη διάσταση του χώρου. Διαισθητικά μπορούμε να υποθέσουμε γιατί συμβαίει αυτό αν αναλογιστούμε πως ο σκοπός είναι διανύσματα που είναι αρκούντως κοντά, να προβάλλονται πάνω στο ίδιο “w-οστό”. Επειδή, είναι γνωστό από τη βιβλιογραφία πως η αναμενώμενη απόσταση δύο διανυσμάτων αυξάνει γραμμικά με το μέγεθος των διαστάσεων (Παράγραφος 2.3, παρατήρηση 5, [1]), αυτό σημαίνει πως για σταθερή τιμή του w, η πιθανότητα δύο διανύσματα να έχουν την ίδια τιμή για κάποια h() μειώνεται με την αύξηση των διαστάσεων. Στις 128 διαστάσεις που αποτελούσαν τον διανυσματικό χώρο των dataset που δώθηκαν, το πρόβλημα αυτό φάνηκε στην πράξη. Για τιμές του w μικρότερες του 100, ελάχιστα διανύσματα είχαν κοινό g(p) μεταξύ τους. Για τις τιμές [200-300] ενώ για κάθε query datapoint ο αριθμός των αποθηκευμένων διανυσμάτων με κοινό g(p) συνήθως ήταν διψήφιος. Επειδή λοιπόν η παράμετρος αυτή εξαρτάται από τη φύση του dataset, το πρόγραμμα υπολογίζει το w ως:
w = floor( t * dim )

όπου dim είναι η διάσταση του χώρου των dataset και t μια real παράμετρος από το χρήστη
που προσδιορίζει πόσο μεγάλη η μικρή διακριτοποίηση θέλουμε να επιτύχουμε κατά την εκτέλεση. Τα εκτελέσιμα μπορούν να κληθούν με το όρισμα -t X ώστε να προσδιορίσουν κατάλληλη τιμή (default 3.0).

  • Τα διανύσματα δεν αποθηκεύονται μέσα στις ίδιες τις δομές αλλά σε ευρετήρια στο top level abstraction του προγράμματος. Οι δομές αναζήτησης αν και χρησιμοποιούν τα διανύσματα για τις σχετικές πράξεις, εντούτοις αποθηκεύουν μόνο τα id τους και αντίστοιχα επιστρέφουν τα κατάλληλα για κάθε query datapoint.

Σύγκριση LSH - Hypercube Projection

Όπως ζητήθηκε, εκτελέστηκαν και οι δύο τρόποι στο ίδιο dataset (shift-small, με την ευκλείδεια νόρμα ως μετρική) με τις κατάλληλες παραμέτρους ώστε να επιτύχουν και οι δύο το ίδιο μέγιστο κλάσμα προσέγγισης. Στη συνέχεια παραθέτονται αποτελέσματα και μετρήσεις από την εκτέλεσή τους:

+--------------------------------------------------------+---------+----------+
| Method                                                 |   LSH   | Hypercube|
+--------------------------------------------------------+---------+----------+
|                                                        | k = 3   | k' = 9   |
| Parameters                                             | L = 3   | probes=2 |
|                                                        | w = 380 | w = 473  |
+--------------------------------------------------------+---------+----------+
| Maximum approximation ratio                            | 2.54    |  2.56    |
| Mean approximate Nearest Neighbor time (ms)            | 8.87    |  2.27    |
|--------------------------------------------------------+---------+----------+
| Memory usage (structures only)(MiB) (*)                | 2.6     |  0.5     |
|--------------------------------------------------------+---------+----------+
| Mean approximation ratio                               | 1.11    |  1.27    |
| Mean number of vectors searched exhaustively per query | 685.93  |  305.18  |
| Number of true Nearest Neighbors found                 | 48      |  28      |
+--------------------------------------------------------+---------+----------+

Όπως διαπιστώνουμε, η προβολή στον υπερκύβο πέτυχε σχεδόν ολόιδιο αποτέλεσμα ως προς το μέγιστο σφάλμα προσέγγισης μειώνοντας τον χρόνο ανά query point στο 1/4 και τη μνήμη που χρειάζεται η αποθηευτική δομή στο 1/5. Μάλισα, το LSH κατέληξε να ψάχνει τα διπλάσια datapoints ανά query.
Ωστόσο να θέλαμε να αναφέρουμε πως το παραπάνω συμπέρασμα δε φανερώνει όλη την αλήθεια για τη σύγκριση των δύο μεθόδων. Αρχικά, δεν μπόρεσα να εκτελέσω τα προγράμματα πολλές φορές και να βγάλω μέσους όρους διότι ήταν πολύ δύσκολο να βρεθεί ξανά κοινό κλάσμα προσέγγισης. Έπειτα, δεν φαίνονται δύο σημαντικά πλεονεκτήματα του LSH έναντι του Hypercube: α) ότι εύκολα μπορεί να μειώσει αρκετά το maximum approximation ratio (ώστε να τείνει στο 1) και β) ότι έχει μικρότερη διακύμανση στα αποτελέσματα ανά εκτέλεση.

(*): Η μέτρηση της μνήμης έγινε με τη χρήση του εργαλείου massif του valgrind. Η χρήση μνήμης των δομών αποθήκευσης συγκεκριμένα έγινε μετρώντας το memory usage του κάθε εκτελέσιμου μέχρι το σημείο της εισαγωγής των διανυσμάτων στις δομές και ύστερα αφαιρώντας τα προϋπολογισμένα overheads (συνολικού χώρου 10.6 MiB) για τις λοιπές λειτουργίες του προγράμματος. Κατά αυτόν τον τρόπο δεν υπολογίζεται στον πίνακα ο χώρος που απαιτείται για να αποθηκευτούν στα διανύσματα (10.24 MiB για το dataset στο σύστημά μου), αφού αρχικά δεν αποθηκεύονται μέσα στις ίδιες τις δομές αλλά σε ευρετήριο κι επίσης επειδή δεν έχει νόημα ο υπολογισμός τους (κάθε μέθοδος πρέπει να αποθηκεύσει κάπου τα διανύσματα). Λόγω της σχεδίασης και της υλοποίησης επίσης (pointers, indexes, references, STL classes, κλπ) δεν είχε νόημα να χρησιμοποιηθεί και η συνάρτηση/τελεστής sizeof().
